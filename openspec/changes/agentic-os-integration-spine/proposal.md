# Agentic OS Integration Spine

## Why

Djimitflo already has worker leases, resource-aware scheduler gates, specialist panels, OKF-backed knowledge, capability sync, learning closure and Mission Control evidence. The missing product-level slice is one integration spine that turns external events into the same governed runtime chain every time.

This change makes `work_items` the canonical integration inbox and proves the OS path:

`external event -> work_item -> goal -> loop -> specialist/skill context -> worker pool -> checker -> eval -> reflection/memory candidate -> dashboard evidence`

## What Changes

- Add a canonical integration intake contract that normalizes GitHub, Telegram, MCP drift, OKF drift and dashboard operator events into existing `work_items`.
- Gate integration influence through existing `swarm_capabilities`, `mcp_servers`, `mcp_tools` and `mcp_tool_permissions`.
- Add an operator-safe run chain that plans selected work items into goals, loop runs and maker/checker leases without auto-starting workers.
- Prove scheduler-driven maker/checker execution, learning closure and memory/reflection candidate creation for integration-origin work.
- Add Mission Control visibility for the source event through learning closure.

## Guardrails

- No new agent platform.
- No separate integration task table for v1.
- No auto-merge, auto-deploy or automatic memory promotion.
- No unattended high-risk execution.
- Worker starts stay behind existing resource, policy and scheduler gates.

## Success Criteria

- OpenSpec validates strictly.
- Goal batch preview is deterministic and read-only.
- External events become idempotent `work_items`.
- Selected work item can be planned into goal, loop and maker/checker leases.
- Scheduler, worker, checker, eval and learning closure evidence link back to the source event.
- Mission Control shows the integration spine without requiring raw stdout inspection.
